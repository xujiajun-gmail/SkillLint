# 项目重点问题

基于当前项目代码与文档，识别并整理项目的「重点问题」：技术难点、设计决策、待解决问题、风险点或核心议题等。

## 使用时机

- 用户希望梳理「重点问题」「技术难点」「待解决问题」「项目风险」「设计中的关键决策」
- 用户提供格式（如「问题/影响/解决方案」表格、Q&A、P0/P1/P2 分级），希望按格式生成
- 用于毕设答辩「存在的问题与改进」、项目汇报「风险与难点」、技术文档「已知问题」

## 总体思路

1. **明确范畴**：技术难点、设计权衡、已知 Bug/限制、性能风险、安全考虑，或待讨论核心议题
2. **从项目中识别依据**：代码、注释、README、TODO/FIXME、Issue、文档
3. **按用户格式组织**：表格、Q&A、分级，或默认分块列表

## 工作流程

1. **确认范围与格式**：解析用户口径（只要技术难点/待解决 Bug/含设计决策）及格式
2. **从仓库识别**：TODO/FIXME、Known issues、Limitations、Future work、设计说明、非常规依赖、单点瓶颈等
3. **分类与归纳**：技术难点、设计决策、待解决问题、风险与限制；合并重复，每类 3–7 条
4. **按格式生成**：表格按列填写、Q&A 格式、分级归入 P0/P1/P2，或分块列表
5. **注明依据**：说明信息来自哪些位置，不虚构无依据的「问题」

## 注意事项

- 有据可依：每条尽量能在代码/注释/文档中找到对应
- 表述客观：使用「当前实现存在……」「README 中标注了……」等
- 与「项目整体梳理」区别：本 Skill 聚焦问题/难点/风险/决策；整体梳理侧重概述、模块、技术选型
