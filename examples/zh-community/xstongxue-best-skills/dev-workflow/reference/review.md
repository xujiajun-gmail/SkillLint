# 代码审查

开发流程第四步。对实现的代码进行质量检查，发现潜在问题并提出改进建议，为后续 Bug 修复提供输入。

## 使用时机

- 用户提到「代码审查」「Code Review」「检查代码」「看看有没有问题」
- 用户已完成代码实现，准备进行质量检查
- 作为开发流程的第四步被调用（代码实现之后，Bug 修复之前）

## 流程串联

```
需求理解 → 方案设计 → 代码实现 → [代码审查] → Bug 修复
                                    ↑ 当前
```

- 上游输入：来自代码实现的实现代码
- 下游输出：审查报告与改进建议，严重问题交给 Bug 修复处理

## 工作流程

### Step 1：审查范围确认

首先明确审查范围：
- 审查哪些文件/模块
- 重点关注哪些方面（功能正确性、性能、安全等）
- 是否有特定的编码规范需要遵循

### Step 2：代码阅读与理解

在提出问题前，先理解代码：
- 整体结构与模块划分
- 数据流与控制流
- 关键业务逻辑

### Step 3：多维度审查

从以下维度进行审查：

**正确性审查**
- 逻辑是否正确，能否实现预期功能
- 边界条件是否处理（空值、越界、异常输入）
- 并发场景是否安全（如适用）

**安全性审查**
- 输入校验是否充分（SQL 注入、XSS 等）
- 敏感信息是否暴露（密码、密钥、Token）
- 权限控制是否完善

**性能审查**
- 是否存在明显的性能瓶颈
- 循环中是否有不必要的重复计算
- 数据库查询是否高效（N+1 问题、缺少索引）

**可维护性审查**
- 命名是否清晰、一致
- 函数/类职责是否单一
- 是否有足够的注释说明复杂逻辑
- 代码是否便于测试

**规范性审查**
- 代码风格是否一致
- 是否遵循项目约定的规范
- 文件组织是否合理

### Step 4：问题分类与标注

将发现的问题按严重程度分类：

| 级别 | 标识 | 含义 | 处理要求 |
|------|------|------|----------|
| 严重 | 🔴 | 必须修复，存在功能错误或安全漏洞 | 修复后才能上线 |
| 建议 | 🟡 | 建议修改，影响可维护性或有潜在风险 | 推荐修复 |
| 优化 | 🟢 | 可选优化，能让代码更好但非必须 | 视情况处理 |

### Step 5：输出审查报告

按以下模板输出审查报告：

```

### 审查概要
- 审查文件：xxx
- 审查时间：xxx
- 整体评价：[简要评价代码质量]

### 问题清单

**问题1：[问题标题]**
- 位置：`文件路径:行号`
- 问题：[问题描述]
- 风险：[可能造成的影响]
- 建议：[修改建议]

```原代码

```

```建议修改为

```

### Step 6：修复指导

对于严重问题和建议修改项：
- 提供具体的修改代码示例
- 解释为什么这样改更好
- 如有多种修改方案，说明各自优劣

### Step 7：复审（如需要）

如果用户修改后希望再次审查：
- 聚焦于修改的部分
- 确认问题是否已正确修复
- 检查修改是否引入新问题

## 常见问题检查清单

**JavaScript/TypeScript**
- [ ] 是否有未处理的 Promise reject
- [ ] 是否有潜在的内存泄漏（事件监听未移除）
- [ ] 类型定义是否完善

**Python**
- [ ] 是否有未关闭的文件/连接
- [ ] 异常处理是否过于宽泛（bare except）
- [ ] 可变默认参数问题

**数据库相关**
- [ ] SQL 是否使用参数化查询
- [ ] 是否有 N+1 查询问题
- [ ] 事务边界是否正确

**API 接口**
- [ ] 输入参数是否校验
- [ ] 错误响应是否规范
- [ ] 是否有适当的日志记录

## 注意事项

- 审查态度应建设性，指出问题的同时给出解决方案
- 区分「必须修改」和「个人偏好」，不强推非必要的风格变更
- 如代码整体质量较好，也要给予肯定
- 审查是为了提升代码质量，而非批评开发者
