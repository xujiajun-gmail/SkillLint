# 文档模板

根据任务类型选择对应模板，按需增删章节。

---

## 周报

**写作要点：**
- 「本周完成」每项写"做了什么 + 结果/产出"，不只写动作
- 数据优先：时间、数量、错误率、完成比例等量化信息必须保留
- 多个同类工作（如模型测试）用表格呈现，不用列表堆叠
- 「问题与风险」写清楚：现象 → 根因 → 影响范围 → 当前应对

```markdown
# 周报 | [姓名/部门] | [日期范围]

## 一、本周完成

### [业务模块 1]

[1-2 句背景说明，说明这块工作的目标或背景]

| 事项 | 结果/产出 | 备注 |
| ---- | ---- | ---- |
| 事项 1 | 具体结果（含数据） | 可选说明 |
| 事项 2 | 具体结果（含数据） | 可选说明 |

> 结论或推荐（如有）：用引用块突出关键判断

### [业务模块 2]

- **事项名称**：做了什么，结果如何（含量化数据）
- **事项名称**：做了什么，结果如何

## 二、进行中

| 事项 | 当前进度 | 预计完成时间 |
| ---- | ---- | ---- |
| 事项 1 | XX% / 阶段描述 | MM月DD日 |
| 事项 2 | 进行中 | MM月DD日 |

## 三、下周计划

- **目标 1**：具体产出或里程碑（预计 MM月DD日）
- **目标 2**：具体产出或里程碑（预计 MM月DD日）

## 四、问题与风险（可选）

- **问题 1**：现象描述 → 根因 → 影响范围 → 当前应对/待决策事项
- **问题 2**：现象描述 → 根因 → 影响范围 → 当前应对/待决策事项
```

**示例片段：**

```markdown
### AI 试做——模型测试

完成 Qwen3.5 系列 6 款模型在 NVIDIA 4×24G 环境下的高考模拟卷测试（共 18 题）。

| 模型 | 速度 (tokens/s) | 用时 | 错题数 | 失败题数 |
| ---- | ---- | ---- | ---- | ---- |
| Qwen3.5-122B-A10B-AWQ | 60–80 | 30 分钟 | 1 | 1（概率性）|
| Qwen3.5-35B-A3B-INT8 | 220 | 25 分钟 | 2 | 2 |

> 推荐：NVIDIA 4×24G 环境首选 **Qwen3.5-122B-A10B-AWQ**，精度与速度综合最优。
```

## 工作汇报

**写作要点：**
- 开头「概述」先给结论，再展开细节（结论前置）
- 「工作进展」用状态标签（✅ 已完成 / 🔄 进行中 / ⏳ 待启动）一目了然
- 「关键成果」强调业务价值，不只是技术细节
- 「需协调事项」要具体说明需要谁、做什么、截止时间

```markdown
# 工作汇报 | [主题] | [日期]

## 一、概述

[2-3 句话：本次汇报范围 + 核心结论 / 关键判断]

## 二、工作进展

| 事项 | 状态 | 关键数据/说明 |
| ---- | ---- | ---- |
| 事项 1 | ✅ 已完成 | 具体产出或结果 |
| 事项 2 | 🔄 进行中（XX%） | 当前阶段说明 |
| 事项 3 | ⏳ 待启动 | 预计启动时间 |

## 三、关键成果

- **成果 1**：[业务价值描述，含量化指标]
- **成果 2**：[业务价值描述，含量化指标]

## 四、下一步计划

| 计划 | 负责人 | 预计时间 |
| ---- | ---- | ---- |
| 计划 1 | — | MM月DD日 |
| 计划 2 | — | MM月DD日 |

## 五、需协调事项（可选）

- **事项**：需要 [谁] 提供 [什么支持]，截止 [时间]
```

## 总结

**写作要点：**
- 「背景与目标」说清楚"为什么做"，不只是"做了什么"
- 「成果与亮点」优先写可量化的成果，再写定性亮点
- 「不足与改进」诚实具体，改进方向要可执行，不写"加强沟通"类空话
- 「经验与启示」提炼可复用的方法论，对后续工作有指导价值

```markdown
# [主题] 总结 | [时间范围]

## 一、背景与目标

[说明总结背景：为什么做这件事、目标是什么、范围边界]

## 二、主要工作

| 工作项 | 产出/结果 | 完成情况 |
| ---- | ---- | ---- |
| 工作 1 | 具体产出 | ✅ / 🔄 / ❌ |
| 工作 2 | 具体产出 | ✅ / 🔄 / ❌ |

## 三、成果与亮点

- **成果 1**：[量化描述，如"将处理时间从 X 降低至 Y"]
- **成果 2**：[定性亮点，说明业务价值]

## 四、不足与改进

| 不足点 | 影响 | 改进方向 |
| ---- | ---- | ---- |
| 不足 1 | 影响描述 | 具体改进措施 |
| 不足 2 | 影响描述 | 具体改进措施 |

## 五、经验与启示（可选）

- **经验 1**：[可复用的方法或教训]
- **经验 2**：[可复用的方法或教训]
```

## 介绍

**写作要点：**
- 「概述」一句话说清楚"这是什么、解决什么问题、给谁用"
- 「核心能力/特点」用表格或分点列出，每点附简短说明，不堆砌技术词汇
- 「适用场景」贴近读者，用具体场景描述，不用抽象定义

```markdown
# [对象] 介绍

## 一、概述

[一句话定位：这是什么 + 核心价值 + 目标用户/场景]

## 二、核心能力

| 能力 | 说明 |
| ---- | ---- |
| 能力 1 | 简要说明，突出亮点 |
| 能力 2 | 简要说明，突出亮点 |

## 三、特点 / 优势

- **特点 1**：具体说明，与同类方案的差异
- **特点 2**：具体说明

## 四、适用场景

- **场景 1**：[具体使用场景描述]
- **场景 2**：[具体使用场景描述]

## 五、联系方式 / 获取方式（可选）

- [链接、联系人、部署方式等]
```
